Stress workload introduction (firmware)

The irregular-timing workload is a bounded irregular-timing stimulus.
It increases timing variability relative to the minimal fixed-period workload, but it is not a broad CPU, memory, flash, or peripheral stress test.

- added single alternate firmware mode: `irregular-timing`
- implemented deterministic irregular TIM3 stimulus intervals
- preserved capture contract: `index,interval_us`
- preserved terminal state: `STATE,CAPTURE_DONE,138`
- default minimal workload unchanged

Constraints enforced:
- irregular-timing is isolated behind feature flag
- compile-time guard prevents combination with divergence injection features
- no changes to CLI, artifact structure, or CSV schema

Build validation:
- irregular-timing firmware builds successfully
- default and stress builds both compile clean
- tests pass
- formatting and diff checks pass

Flashing procedure correction

Initial condition:
- irregular-timing firmware was flashed while target rail was ~1.755 V

Observed:
- st-flash reported low voltage and downgraded to 8-bit flash writes
- flash completed and verified under low-voltage condition

Correction:
- board rail restored to nominal operating voltage prior to flashing
- irregular-timing firmware reflashed under normal conditions

Verification:
- st-flash used 32-bit flash writes at nominal voltage
- flash write completed successfully
- flash-compare-ur passed (device binary matches build artifact)

Interpretation:
- low-voltage flashing is not a valid operating condition
- correct procedure is:
  - flash and verify at nominal voltage
  - perform undervoltage only during execution/capture phase
